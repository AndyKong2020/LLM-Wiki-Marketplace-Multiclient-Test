# Multi-Client Distribution Design

## Context

This repository currently publishes `llm-wiki-client@llm-wiki-cloud` as a Claude Code marketplace plugin for the cloud CANN-Infer-Wiki service. The plugin provides:

- a remote read-only MCP server at `https://wiki.andykong.top/mcp`
- mount flow: probe MCP and write project instructions
- query flow: use wiki search/page tools and record usage
- backflow flow: create local task archive and optionally upload through token-gated HTTP

The next iteration must support Claude Code, Codex, and OpenCode from one repository without requiring maintainers to hand-edit three divergent copies.

## Goals

- Maintain one source of truth for skill behavior, command prompts, endpoint constants, and release version.
- Publish platform-specific adapter files for Claude Code, Codex, and OpenCode.
- Keep installation low friction for all three tools.
- Test complete mount, query, and backflow behavior across all three clients.
- Keep all tests isolated from production remotes and global user configuration.

## Non-Goals

- Do not replace the production `cloud` or `origin` remotes.
- Do not upload test backflow archives to the production backflow endpoint.
- Do not require users to install a local MCP server.
- Do not introduce a dynamic OpenCode JS/TS plugin unless static config, commands, and skills prove insufficient.

## Repository Strategy

Use one editable source tree and generated platform adapters.

```text
VERSION
src/
  shared/
    constants.json
    pin-block.md.tmpl
  skills/
    llm-wiki-cloud-mount/SKILL.md.tmpl
    llm-wiki-cloud-query/SKILL.md.tmpl
    llm-wiki-cloud-backflow/SKILL.md.tmpl
  commands/
    wiki-cloud-mount.md.tmpl
    wiki-cloud-backflow.md.tmpl

platforms/
  claude/
    marketplace.json.tmpl
    plugin.json.tmpl
  codex/
    marketplace.json.tmpl
    plugin.json.tmpl
  opencode/
    opencode.json.tmpl
    install-opencode.sh.tmpl

plugins/llm-wiki-client/
  .claude-plugin/plugin.json
  .codex-plugin/plugin.json
  .mcp.json
  README.md
  commands/
  skills/

dist/opencode/
  opencode.json
  install-opencode.sh
  .opencode/
    commands/
    skills/

scripts/
  sync_adapters.py
  validate_release.py
  test_isolated_clients.py
```

Maintainers edit only `VERSION`, `src/`, `platforms/`, top-level docs, and scripts. Generated files under `plugins/llm-wiki-client/` and `dist/opencode/` are committed for marketplace and copy-paste installation, but must be regenerated by `scripts/sync_adapters.py`.

Generated files should include a short generated-file marker where the target format allows comments.

## Platform Adapters

### Claude Code

Claude Code keeps the current marketplace shape:

```text
.claude-plugin/marketplace.json
plugins/llm-wiki-client/.claude-plugin/plugin.json
plugins/llm-wiki-client/.mcp.json
plugins/llm-wiki-client/commands/*.md
plugins/llm-wiki-client/skills/*/SKILL.md
```

Installation remains:

```text
/plugin marketplace add AndyKong2020/LLM-Wiki-Marketplace-Cloud
/plugin install llm-wiki-client@llm-wiki-cloud
/reload-plugins
```

Mount writes or updates `CLAUDE.md` because that is the native Claude Code project instruction file.

### Codex

Codex reuses the same plugin root, with an additional manifest and marketplace:

```text
.agents/plugins/marketplace.json
plugins/llm-wiki-client/.codex-plugin/plugin.json
plugins/llm-wiki-client/.mcp.json
plugins/llm-wiki-client/skills/*/SKILL.md
```

The Codex adapter must avoid Claude-specific tool names in skill bodies. Skills should refer to the MCP server and logical tool names:

```text
cann-infer-wiki-cloud wiki_search
cann-infer-wiki-cloud wiki_get_page
```

Mount writes or updates `AGENTS.md`.

### OpenCode

OpenCode uses a generated static adapter package:

```text
dist/opencode/opencode.json
dist/opencode/install-opencode.sh
dist/opencode/.opencode/commands/*.md
dist/opencode/.opencode/skills/*/SKILL.md
```

`opencode.json` configures the remote MCP:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "cann-infer-wiki-cloud": {
      "type": "remote",
      "url": "https://wiki.andykong.top/mcp",
      "enabled": true
    }
  }
}
```

OpenCode commands are markdown files in `.opencode/commands/`. OpenCode skills are `SKILL.md` files in `.opencode/skills/<name>/`. Mount writes or updates `AGENTS.md`; OpenCode can fall back to `CLAUDE.md`, but the native rules file is `AGENTS.md`.

## Command and Skill Behavior

### Mount

Mount must:

- read local adapter version from generated skill metadata
- fetch `https://wiki.andykong.top/plugin/llm-wiki-client/version.json`
- stop if local version is older than remote version
- probe `wiki_search(query="mount probe", limit=1)`
- write the latest LLM-WIKI pin block
- report version, MCP mode, probe result, pin status, and target instruction file path

Instruction file target:

| Platform | Target |
|---|---|
| Claude Code | `CLAUDE.md` |
| Codex | `AGENTS.md` |
| OpenCode | `AGENTS.md` |

### Query

Query must:

- build a specific wiki query from the current model/kernel/framework/optimization context
- call `wiki_search`
- call `wiki_get_page` for relevant candidates
- download assets before inspecting images
- record every fetched page in `wiki_usage.md`
- append a query summary to `progress.md` when a task progress file exists
- treat wiki content as advisory, not ground truth

### Backflow

Backflow must:

- infer a task slug and workspace
- create `.claude/llm-wiki/backflow/<task-slug>/`
- include exactly one top-level markdown entrypoint
- include task traces, wiki usage, small related files, git status/diff, and relevant agent summaries when present
- exclude credentials, tokens, `.env*`, `.git/`, IDE state, raw profiler dumps, model weights, datasets, caches, and large binaries
- report archive summary before any upload
- upload only after user confirmation and only when `LLM_WIKI_UPLOAD_TOKEN` is set
- support `LLM_WIKI_UPLOAD_URL` for staging or mock upload tests
- never print or archive the token

## Versioning and Release Flow

Release version lives in `VERSION`.

Release steps:

```bash
python3 scripts/sync_adapters.py
python3 scripts/validate_release.py
git diff --exit-code
git tag "v${VERSION}"
```

`sync_adapters.py` must propagate the version to:

- root Claude marketplace
- Claude plugin manifest
- Codex plugin manifest
- Codex marketplace
- all generated skill frontmatter
- plugin README
- generated OpenCode install package

`validate_release.py` must check:

- JSON validity
- all versions match `VERSION`
- all generated files are current
- all MCP URLs match the canonical URL
- OpenCode skill names match `^[a-z0-9]+(-[a-z0-9]+)*$`
- exactly the expected command names exist
- no token-like strings are present in committed files
- README install commands match generated adapter names

## Isolated Test Repository

Use a separate private GitHub repository for multi-client testing:

```text
AndyKong2020/LLM-Wiki-Marketplace-Multiclient-Test
```

Local remote name:

```text
multiclient-test
```

Rules:

- Never push test branches to `origin` or `cloud`.
- Never change the URL for `origin` or `cloud`.
- Test pushes must use explicit remote names:

```bash
git push multiclient-test "$BRANCH"
```

- The test repository may receive generated adapters, test branches, GitHub Actions workflows, and release dry-runs.
- Production marketplace repository remains untouched until test acceptance passes.

## Isolated Local Test Harness

All automated client tests run under a temporary test root.

```bash
LLM_WIKI_TEST_ROOT="$(mktemp -d)"
export HOME="$LLM_WIKI_TEST_ROOT/home"
export XDG_CONFIG_HOME="$LLM_WIKI_TEST_ROOT/xdg-config"
export XDG_DATA_HOME="$LLM_WIKI_TEST_ROOT/xdg-data"
export CODEX_HOME="$LLM_WIKI_TEST_ROOT/codex-home"
export OPENCODE_CONFIG="$LLM_WIKI_TEST_ROOT/opencode/opencode.json"
export OPENCODE_CONFIG_DIR="$LLM_WIKI_TEST_ROOT/opencode"
export LLM_WIKI_UPLOAD_TOKEN="llmw_test_token"
export LLM_WIKI_MOCK_PORT="$(python3 scripts/allocate_test_port.py)"
export LLM_WIKI_UPLOAD_URL="http://127.0.0.1:${LLM_WIKI_MOCK_PORT}/upload/backflow"
```

The harness must create a temporary project workspace inside `LLM_WIKI_TEST_ROOT`, not inside a real user project.

Before and after each test run, the harness must check that these real global paths did not change:

```text
~/.claude
~/.codex
~/.agents
~/.config/opencode
```

If a client ignores the temporary environment and writes to a real global path, the test fails immediately.

## Complete Test Coverage

### Static Validation

- Generate adapters from templates.
- Validate all JSON files.
- Validate skill frontmatter.
- Validate command names and descriptions.
- Validate version consistency.
- Validate no production upload token is present.

### Claude Code Smoke Test

Run with temporary home/config only.

- Install or load the generated Claude plugin from the test repository or local checkout.
- Confirm the remote MCP config is visible.
- Run mount against a temporary project.
- Verify `CLAUDE.md` contains exactly one LLM-WIKI block.
- Run mount twice and verify idempotency.
- Trigger a query task and verify `wiki_usage.md` and `progress.md` are written.
- Run backflow archive creation and verify archive layout.
- Upload to local mock backflow server, not production.

### Codex Smoke Test

Run with temporary `CODEX_HOME`, `HOME`, and `XDG_*`.

- Install or load the generated Codex plugin from the test repository or local checkout.
- Confirm `.codex-plugin/plugin.json`, `.mcp.json`, and skills are discovered.
- Run mount against a temporary project.
- Verify `AGENTS.md` contains exactly one LLM-WIKI block.
- Run mount twice and verify idempotency.
- Trigger a query task and verify `wiki_usage.md` and `progress.md` are written.
- Run backflow archive creation and verify archive layout.
- Upload to local mock backflow server.

### OpenCode Smoke Test

Run with temporary `HOME`, `XDG_*`, `OPENCODE_CONFIG`, and `OPENCODE_CONFIG_DIR`.

- Run `dist/opencode/install-opencode.sh --prefix "$LLM_WIKI_TEST_ROOT/opencode-install"` or equivalent dry-run mode.
- Verify commands, skills, and `opencode.json` are installed only under the test root.
- Verify MCP config contains `cann-infer-wiki-cloud` as a remote enabled server.
- Run mount against a temporary project.
- Verify `AGENTS.md` contains exactly one LLM-WIKI block.
- Run mount twice and verify idempotency.
- Trigger a query task and verify `wiki_usage.md` and `progress.md` are written.
- Run backflow archive creation and verify archive layout.
- Upload to local mock backflow server.

### Mock Backflow Server

The test harness must provide a local HTTP server that accepts multipart uploads and returns:

```json
{
  "status": "ok",
  "id": "test-backflow-id",
  "path": "sources/sessions/uploaded/test",
  "entrypoint": "test.md"
}
```

Additional test cases:

- `duplicate` response is treated as success.
- `error` response is reported and does not delete the local archive.
- package over 50 MiB is rejected before upload.
- missing or multiple top-level markdown files are rejected before upload.
- token is not printed in stdout, stderr, archive files, or git diff.

## Acceptance Criteria

- A maintainer can update content once, run `sync_adapters.py`, and get all three client adapters.
- Claude Code, Codex, and OpenCode can install from the same test repository or generated package.
- Mount, query, and backflow are tested for all three clients.
- Tests do not modify production remotes.
- Tests do not modify real global client configuration.
- Test upload never reaches `https://wiki.andykong.top/upload/backflow`.
- Production release is blocked unless `validate_release.py` and isolated client tests pass.

## Implementation Order

1. Add `VERSION`, source templates, platform templates, and generation script.
2. Generate Claude/Codex shared plugin root.
3. Generate OpenCode adapter package and installer.
4. Add release validator.
5. Add isolated test harness with mock backflow server.
6. Push feature branch only to `multiclient-test`.
7. Run full client smoke tests from the private test repository.
8. After acceptance, separately decide whether and when to promote changes to production remotes.
